# Coordination — `sst-cam-app`

What a change on either side forces the other to do.

## When firmware changes interaction (firmware → app)
- **Credential/handshake format** (e.g. WiFi Direct creds now generated by the
  camera, not pre-shared) → the app must change how it reads/joins. Raise an
  `outbound/` handoff; the app runs its own plan/work cycle against it.
- **Session ordering or new required step** → the app must send the new
  command/sequence. Handoff + acceptance criteria.
- **Overlay rendering semantics** (coordinate space, anchor/alignment, fonts) →
  both sides must agree or previews drift from the recorded/streamed frame.
  Co-development handoff.

## When the app changes interaction (app → firmware)
- A new command the app wants firmware to honor, a changed polling cadence, or a
  new field in a pushed message → arrives as an `inbound/` handoff (or via a
  proto change). Firmware acts, then fills the handoff's `## Response` section.

## Shared invariants to preserve
- App initiates, firmware responds; one response per command, matched by
  `correlation_id`. Firmware never pushes unsolicited data.
- Discovery filter = service UUID **+** `sst-cam-NNNN` name prefix.
- Business data (users/teams/matches/streaming configs/overlay design) lives in
  the app. Firmware re-adding a local business store is a permanent non-goal.

## Both sides depend on `sst-cam-proto`
- Interaction changes that need a new message/field go through the proto repo
  first (see `../proto/coordination.md`), then both app and firmware re-pin.
